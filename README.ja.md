# No Rules Rules Skill

すぐに使える AI スキルパック——ネットフリックスの「自由と責任」文化の組織設計手法を、10個の個別起動・組み合わせ可能なマネジメントモジュールに蒸留しました。

これは本の要約ページではありません。本はこれらのスキルの源泉であり、主役ではありません。

## このスキル群が解決する問題

- 「まあまあ」のメンバーを残すべきかどうか
- フィードバック文化を感情発散にせずに構築する方法
- 承認を減らしつつ責任を失わない方法
- 高い自律性を持つチームで、誰が最終判断するか
- 異文化環境で率直なフィードバックをどう適応させるか

## 何が蒸留されたか

『No Rules Rules』から10個の三重検証済みスキルを抽出し、4つの次元をカバーしています：

- **フィードバックとコミュニケーション**：`4a-feedback`（4Aフィードバック）、`feedback-loop-360`（360度フィードバックループ）、`cross-cultural-feedback`（異文化フィードバック）
- **人材とチーム**：`talent-density`（人材密度）、`keeper-test`（キーパーテスト）
- **マネジメントと意思決定**：`context-not-control`（コンテキスト・ノット・コントロール）、`informed-captain`（インフォームド・キャプテン）
- **組織文化**：`elite-compensation`（エリート報酬）、`radical-transparency`（ラディカル・トランスペアレンシー）、`eliminate-controls`（管理の排除）

各スキルは三重検証を通過しています——真のクロスシーン適用性、予測力、独自性を持つ方法論のみが最終的な成果物になっています。

## 効果例

### 例1：「まあまあ」のメンバーを残すべきか

**ユーザーの質問**

「チームに悪くはないけれど、キーパーソンではないメンバーがいます。残すべきでしょうか？」

**スキルの判断**

- まず `keeper-test` を起動——この人は積極的に引き留める人か？
- 問題がチームの質に拡大する場合は `talent-density` に接続
- 報酬戦略に関連する場合は `elite-compensation` に接続

**回答例**

> まず「明確な失敗をしたか」ではなく、より厳しい質問をしてください：明日辞めると言われたら、本気で引き留めようとするか？答えがノーなら、問題は通常「基準を満たしているか」ではなく「その席を占め続ける価値があるか」です。次に人材密度を見てください。平均的なパフォーマーは平均的なチームでは問題なく見えますが、高密度チームではフィードバックの速度、判断の質、実行の基準を引き下げます。強力に引き留めないと分かっているなら、曖昧な評価で先延ばししないでください。

### 例2：ルールを削減する前に

**ユーザーの質問**

「会社の承認が多すぎて、みんな遅いと不満を言っています。ルールを全部無くすべきですか？」

**スキルの判断**

- まず `context-not-control` を起動——本当の問題はコンテキスト不足か？
- 次に `eliminate-controls` に接続——どのルールを原則に置き換えられるか？
- 最終判断者を決める必要があれば `informed-captain` に接続

**回答例**

> 「ルールを削減する」ことから始めないでください——まずチームが十分なコンテキストを共有しているか確認してください。コンテキストがない状態で承認をなくすと、混乱が早まるだけです。まず3つを確認する必要があります：全員が十分な情報を持っているか、全員が同じ目標を理解しているか、全員が判断の境界を知っているか。この3つが揃って初めて、ルールを原則に置き換え、事前承認を事前のコンテキスト設定と事後レビューに変えることができます。この3つが揃っていないなら、あなたが削っているのは官僚主義ではなく、最後のエラーキャッチメカニズムです。

## これらのスキルはどのように生成されたか

これらのスキルは **cangjie-skill** の RIA-TV++ パイプラインを使用して生成されました。6段階のプロセスです：

1. **全書理解（Adler分析）** — モーティマー・アドラーの分析的読書法で、全書を構造・解釈・批判・応用の4ステップで分解し、`BOOK_OVERVIEW.md` を生成
2. **並行抽出** — 5つの専門エクストラクター（フレームワーク、原則、事例、反例、用語）が同時に実行され、原文から候補ユニットを抽出
3. **三重検証** — 各候補は3つのチェックを通過する必要があります：書中に少なくとも2つの独立した裏付けがあるか（クロスドメイン）、書中に明示されていない新しい質問に答えられるか（予測力）、常識ではないか（独自性）。合格率は通常25〜50%
4. **RIA++ 構築** — 検証済みの内容を6つの次元に構造化：R（原文引用）/ I（自分の言葉での再構築）/ A1（書中の事例）/ A2（将来のトリガーシーン）/ E（実行可能ステップ）/ B（境界と盲点）
5. **ツェッテルカステン連携** — スキル間の依存、対比、構成関係を特定し、参照グラフ付きの `INDEX.md` を生成
6. **ストレステスト** — 各スキルに囮問題を含むテストプロンプトを設計。不合格は段階4に戻って全面的に再構築

各 `SKILL.md` はこのプロセスの実際の出力です。`candidates/` と `rejected/` ディレクトリは完全な監査証跡として保持されています。

## 使い方

1. `INDEX.md` を参照してスキル全体図と依存関係を把握する
2. 現在の問題に関連する `*/SKILL.md` を見つけ、トリガー条件と実行ステップを使用する
3. エージェントフレームワークに統合するか、プロンプトを単独ツールとして使用する
4. `test-prompts.json` で検証——正しいシーンでトリガーされ、間違ったシーンで沈黙することを確認
